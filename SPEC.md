## 1. Architecture Overview

The plugin follows a strict unidirectional data flow:

```
VaultIndex  →  SimplicialModel  →  LayoutEngine  →  Renderer
               (source of truth)   (forces)          (projection)
                      ↑
               InteractionController
```

### Module Responsibilities

| Module | Responsibility | Dependencies |
|--------|---------------|--------------|
| `core/` | Pure data model, types, normalization | None (vanilla TypeScript) |
| `data/` | Vault indexing, parsing, persistence, inference | Obsidian API |
| `layout/` | Force simulation, Barnes-Hut optimization | None (pure TS) |
| `render/` | Canvas rendering (blobs, edges, nodes) | Browser APIs |
| `interaction/` | User input handling, focus state | Model, browser APIs |
| `ui/` | Obsidian views, modals, panels | Obsidian API |

### Key Invariant

**`SimplicialModel` has zero Obsidian API dependencies.** It is pure TypeScript and fully unit-testable in isolation. The renderer is a projection and contains no business logic. Interaction perturbs the layout; it never rebuilds it.

## 2. Layering Principles

### 2.1 Separation of Concerns

- **Model** owns simplex data, node positions, and lifecycle states
- **Layout** computes forces; does not render
- **Renderer** draws pixels; does not compute business logic
- **Interaction** reads model state, perturbs layout, triggers re-renders
- **Persistence** reads/writes vault files; updates model

### 2.2 Data Flow Rules

1. Vault changes flow **down**: Index → Model → Layout → Renderer
2. User actions flow **up**: Interaction → Model updates → Layout wake → Renderer refresh
3. No module may skip layers (e.g., Renderer must not directly modify Model)
4. Settings are the only cross-cutting concern; passed explicitly to constructors

### 2.3 Purity Constraints

- `core/model.ts` must remain import-free from Obsidian
- `layout/engine.ts` must remain import-free from Obsidian and DOM APIs
- Test files in `tests/` should target `core/` and `data/inference/` first

## 3. Data Model

### 3.1 Core Types

```typescript
type NodeID = string;          // Vault file path (e.g., "Ideas/alpha.md")
type SimplexKey = string;      // Canonical key: "a.md|b.md|c.md" (sorted, lowercase)

interface Simplex {
  nodes: NodeID[];             // Member node IDs
  weight?: number;             // 0.1–1.0 cohesion strength
  label?: string;              // Human-readable name
  colorKey?: ColorKey;         // Deterministic color by simplex order
  autoGenerated?: boolean;     // True if face of higher-order simplex
  userDefined?: boolean;       // True if explicitly created
  inferred?: boolean;          // True if created by inference engine
  source?: SimplexSource;      // Origin: user/inferred/suggested
  sourcePath?: string;         // Vault file that owns this simplex
  lifecycleState?: SimplexLifecycleState;  // weak → explored → confirmed → labeled → promoted
}
```

### 3.2 Simplex Dimensions

- **0-simplex**: Single note (node)
- **1-simplex**: Coherent pair (edge)
- **2-simplex**: Coherent triple (cluster)
- **3-simplex**: Coherent quadruple (core)
- **n-simplex**: n+1 notes forming a coherent unit

### 3.3 Face Generation

When a simplex is defined, all sub-faces are auto-generated up to dimension 4 (5 nodes). Auto-generated faces render more faintly and cannot be directly edited.

**Cap**: Face generation stops at dimension 4 to prevent combinatorial explosion. Higher-order simplices store faces lazily on demand.

### 3.4 Canonical Key Computation

```
normalizeKey(["Talent", " STARTUP ", "capital"])
  → lowercases → ["talent", " startup ", "capital"]
  → trims → ["talent", "startup", "capital"]
  → sorts → ["capital", "startup", "talent"]
  → joins with "|" → "capital|startup|talent"
```

## 4. Layout Engine

### 4.1 Force Simulation

The layout uses a force-directed simulation with:

| Force | Purpose | Configurable |
|-------|---------|--------------|
| **Repulsion** | All nodes repel (Barnes-Hut O(n log n)) | `repulsionStrength` |
| **Spring** | Connected nodes attract | `cohesionStrength` |
| **Simplex cohesion** | Nodes in same simplex cluster toward centroid | `cohesionStrength` |
| **Gravity** | Gentle pull toward origin | `gravityStrength` |
| **Noise** | Organic breathing motion | `noiseAmount` |
| **Hold repulsion** | Push neighbors apart on long-press | Fixed (900) |

### 4.2 Barnes-Hut Optimization

- Quad-tree construction for O(n log n) repulsion calculation
- Theta parameter (default 0.65) controls approximation accuracy
- Direct O(E) calculation retained for spring forces (edges are sparse)

### 4.3 Sleep Mode

The render loop pauses when average kinetic energy falls below `sleepThreshold`. Wakes on:
- User interaction
- Vault changes
- Model updates

### 4.4 Sparse Graph Detection

When the graph contains only 1-simplices (edges) and inferred higher-order simplices, the engine applies:
- Increased edge length spacing (`sparseEdgeLength`)
- Boosted gravity (`sparseGravityBoost`)
- Structural weighting based on node connectivity

## 5. Renderer

### 5.1 Dual View Modes

| Mode | Description |
|------|-------------|
| **Organic (default)** | Soft metaball blobs, overlapping fields, ambient feel |
| **Formal** | Crisp geometric rendering with wireframe edges |

### 5.2 Rendering Pipeline

1. **Viewport culling**: Only render nodes within visible area
2. **Progressive loading**: Incrementally increase detail based on zoom/pan
3. **Blobs**: Capsule-union metaball approach for 2+ simplices
4. **Edges**: Wireframe lines for 1-simplices
5. **Nodes**: Circles with radius based on connectivity
6. **Labels**: Conditional rendering based on focus state and `labelDensity`

### 5.3 Performance Optimizations

- **Text width caching**: Avoids repeated `measureText()` calls
- **Progressive scene key**: Reduces detail when camera is moving
- **Viewport padding**: Pre-loads nearby nodes for smooth panning
- **Last-rendered set**: Avoids redundant draw calls

### 5.4 Color Palette

Deterministic colors by simplex order with stable per-simplex variation. Six-color palette: purple, teal, coral, pink, blue, amber.

## 6. Interaction Model

### 6.1 Core Principle

**Interaction reveals structure, it does not manipulate it.**

### 6.2 Action Mapping

| Action | Effect |
|--------|--------|
| Hover node | Focus mode — simplex fields intensify, unrelated nodes fade |
| Move away | Focus releases with 150ms fade |
| Click-and-hold | Momentary repulsion — push overlapping neighbors apart |
| Double-click | Pin/unpin node — fixes position across sessions |
| Toggle `1`/`2`/`3` | Show/hide edges, clusters, cores |
| `F` | Lock focus on hovered node until Escape |
| `P` | Open metadata panel for hovered simplex |
| `Escape` | Clear all focus and selection |
| Lasso drag | Select multiple nodes for simplex creation |

### 6.3 Focus State

Focus is non-destructive:
- Hovering reveals structural context
- Locking (`F`) persists focus until explicitly cleared
- Focus does not modify the model; only display alpha values

## 7. Persistence

### 7.1 Modes

| Mode | Description |
|------|-------------|
| **source-note** (default) | Simplices stored in YAML frontmatter of member notes |
| **central-file** | All simplices in `_simplicial.md` |

### 7.2 Source-Note Format

```yaml
---
simplices:
  - nodes: [startup, capital, talent]
    label: "founding engine"
    weight: 0.9
---
```

### 7.3 Write Strategy

1. Parse file content
2. Apply simplex updates to frontmatter
3. Modify file via Obsidian Vault API
4. Index records the write to prevent re-parsing

### 7.4 Rename Tracking

When a note is renamed:
1. Model updates all simplex memberships
2. Layout positions are preserved
3. Source paths are updated in vault files on next write

## 8. Edge Inference

### 8.1 Signal Sources

| Signal | Weight | Description |
|--------|--------|-------------|
| Outbound link | `linkWeight` (0.25) | Explicit note-to-note link |
| Mutual link bonus | `mutualLinkBonus` (0.1) | Both notes link each other |
| Shared tag | `sharedTagWeight` (0.15) | Per shared tag |
| Title overlap | `titleOverlapWeight` (0.2) | Token overlap in titles |
| Content overlap | `contentOverlapWeight` (0.15) | Token overlap in body |
| Same folder | `sameFolderWeight` (0.1) | Direct folder match |
| Same top folder | `sameTopFolderWeight` (0.05) | Top-level folder match |

### 8.2 Inference Pipeline

1. **Collect**: Gather all notes with tokens, tags, links, folders
2. **Score**: Compute pairwise signal strength
3. **Threshold**: Create edges above `inferenceThreshold`
4. **Suggest**: Detect triangle closures and soft clusters
5. **Confidence**: Assign confidence score based on signal diversity

### 8.3 Temporal Decay

Older simplices lose strength over time:
- Half-life: `decayHalfLifeDays` (default 90)
- Minimum weight: `decayMinimumWeight` (default 0.1)
- Decay applied during full scan

## 9. Critical Implementation Constraints

These constraints **must** be preserved in all future development:

### 9.1 Model Purity

**`SimplicialModel` must never import from `obsidian`.** It is pure TypeScript and must remain unit-testable in isolation. Any Obsidian-specific logic belongs in `data/` or `ui/`.

### 9.2 Renderer as Projection

**The renderer must not contain business logic.** It reads model state and layout positions; it does not compute simplices, inference, or persistence. If you find yourself adding logic to `renderer.ts`, ask: "Should this be in the model or interaction layer?"

### 9.3 Vault as Source of Truth

**Never store user data outside the vault.** Settings (pins, filters, UI state) go in plugin storage. Simplex definitions (nodes, labels, weights) go in vault files. The vault must survive plugin reinstallation without data loss.

### 9.4 No Silent Failures

**All file I/O errors must be logged and surfaced.** Use `logger.warn()` or `logger.error()` for recoverable failures. Show `Notice` for user-facing failures. Never swallow exceptions silently.

### 9.5 Face Generation Cap

**Face generation must not exceed dimension 4.** Higher-order simplices are stored but faces are computed lazily. This prevents combinatorial explosion (a 10-node simplex has 2^10 - 1 = 1023 faces).

### 9.6 Canonical Ordering

**Simplex keys must always be sorted and lowercased.** This ensures `["A", "B"]` and `["b", "a"]` produce the same key. Violating this causes duplicate simplices and broken references.

## 10. Performance Requirements

| Metric | Target | Notes |
|--------|--------|-------|
| **Initial load (<500 notes)** | <2s | Full scan + render |
| **Initial load (>500 notes)** | <5s | Progressive rendering acceptable |
| **Frame rate (idle)** | 60fps | When awake |
| **Frame rate (interaction)** | 30fps minimum | During drag/pan/zoom |
| **Memory usage** | <100MB | For vaults up to 5000 notes |
| **Sleep mode CPU** | <1% | When layout is asleep |
| **Bundle size** | <500KB | `main.js` production |

### 10.1 Optimization Strategies

- Barnes-Hut quad-tree for O(n log n) repulsion
- Viewport culling for render efficiency
- Progressive loading for large vaults
- Text measurement caching
- Debounced settings saves (150ms)
- Scheduled rescans (50ms delay)

## 11. Testing Strategy

### 11.1 Test Pyramid

```
        /  E2E  \          Manual vault testing (future)
       / Integration\       VaultIndex + Inference (future)
      /    Unit       \     core/model, core/normalize, data/inference
     /   Regression     \   Rename tracking, persistence round-trips
```

### 11.2 Current Coverage

- ✅ `normalizeKey` canonicalization
- ✅ `hashLabel` determinism
- ✅ `generateFaces` expansion
- ✅ `getFacesLazy` dimension capping
- ✅ Model rename + pinning
- ✅ Inference engine (multiple scenarios)
- ✅ Model analysis summary

### 11.3 Missing Coverage (Priority Order)

1. **Layout engine** — Force correctness, sleep/wake cycles
2. **Persistence** — Read/write round-trips for both modes
3. **Parser** — Edge cases (special chars, nested YAML, aliases)
4. **VaultIndex** — Change detection, rename tracking
5. **Renderer** — Viewport culling, zoom/pan behavior

### 11.4 Test Execution

```bash
npm run check   # TypeScript type check
npm test        # Compile and run tests
```

Tests use Node's native `node:test` and `assert` modules. Output goes to `tests-dist/` before execution.

## 12. Build & Deployment

### 12.1 Build Process

```bash
npm run build       # Production (minified, no sourcemaps)
npm run build:dev   # Development (inline sourcemaps, no minify)
```

Build tool: **esbuild** (not webpack). Configuration in `build.mjs`.

### 12.2 Output Files

| File | Purpose |
|------|---------|
| `main.js` | Bundled plugin code (generated) |
| `manifest.json` | Plugin metadata (hand-edited) |
| `styles.css` | Plugin styles (hand-edited) |

### 12.3 Installation

Copy `main.js`, `manifest.json`, `styles.css` to `.obsidian/plugins/simplicial-complex/` in vault.

### 12.4 CI/CD

GitHub Actions workflow (`.github/workflows/ci.yml`) runs on push/PR:
1. Lint (ESLint)
2. Format check (Prettier)
3. Type check (TypeScript)
4. Build verification
5. Test suite
